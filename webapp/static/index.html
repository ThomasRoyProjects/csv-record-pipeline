Normalization is the safest place to remove superficial differences before matching, including unit-aware address cleanup.
  • Use profiles to join split address fields, copy one source column into a target field, or coalesce fallback columns.
  • Built-in address normalization also tries to split apartment-style addresses like 101-1232 Main Street into primary_address1=1232 Main Street and primary_address2=101.
  • Use strict cleanup when you want text standardized for comparison by trimming whitespace, collapsing repeated spaces, and forcing case-insensitive text.
  • Run normalization first when messy source exports would otherwise create false “new” people.

Custom Normalization Profile

          

          
Build reusable profiles for weird imports, such as addresses split across multiple columns or fallback fields spread across inconsistent source sheets.

          

            

              `copy`
              Use one source column as-is.
            

            

              `join`
              Combine several source columns into one field, such as street number plus street name.
            

            

              `coalesce`
              Take the first non-empty value from several possible source columns.

            Advanced Run Settings
            

              

                Quick presets
                Pick a starting posture instead of editing raw numbers first.
              

              

                
                
                
              

            

            

              

                

                  
Output directory

                  
This is the destination for everything the run writes.

                  
    
                    
  • Use it when you want this run isolated from older outputs.
    • 
                    
  • Keep one folder per job or per investigation if you want cleaner review later.
    • 
                  

                

                

                  
Strict match mode

                  
This makes the matcher more conservative.

                  
    
                    
  • Use it when bad matches would be costly.
    • 
                    
  • Leave it off when you want the normal scoring behavior and are willing to review gray-area rows.
    • 
                  

                

                

                  
Thresholds

                  
These numbers control how much evidence is needed before a row moves into a stronger match bucket.

                  
    
                    
  • Confident: score needed to call a row a clear match.
    • 
                    
  • Possible: score range where the matcher sees meaningful evidence but not enough for a confident call.
    • 
                    
  • Review: the lowest score that still gets surfaced for human inspection instead of being treated as unmatched.
    • 
                  

                

                

                  
Score scale

                  
The matcher uses a few-hundred-point score, not a 1 to 10 scale and not giant numbers like 100000.

                  
    
                    
  • 0 to 80: weak evidence or no reliable match support.
    • 
                    
  • 85 to 119: low-confidence review territory.
    • 
                    
  • 120 to 159: stronger possible-match territory.
    • 
                    
  • 160+: strong evidence and eligible for confident classification when the rule checks agree.
    • 
                    
  • Typical strong exact-ID or name-plus-address cases often land somewhere in the 160 to 250+ range.
    • 
                  

                

                

                  
Default advice

                  
    
                    
  • Do not tune thresholds first. Fix mappings and normalization first.
    • 
                    
  • If the review bucket is too large, that usually means the inputs are messy or the mappings are incomplete.
    • 
                    
  • If you only want safe new people, be careful about lowering thresholds too aggressively.
    • 
                  

                

              

            

            

              

                
Match Thresholds

                
Tune how aggressive the matcher should be when assigning confident, possible, and review decisions.

              

              

                
                
                
              

              
Reference scale: exact-ID support alone is worth about 130 points, exact full normalized address is worth about 70, exact email about 45, and exact phone about 35. That is why values like 85, 120, and 160 are meaningful here.

Custom Matching Help

          
        

        

          
`primary` is the file you want to classify. `reference` is the file you trust as the comparison baseline.

          
Use `custom_job` when presets are too specific. The safest default stage order is:

          
    
            
  1. `normalize_addresses` on `primary`
    2. 
            
  2. `normalize_addresses` on `reference`
    3. 
            
  3. `match_records`
    4. 
            
  4. `write_records_bundle`
    5. 
          

          
Map these first:

          
    
            
  • `person_id` if a shared ID exists
    • 
            
  • `first_name`
    • 
            
  • `last_name`
    • 
            
  • `primary_address1`
    • 
          

          
Add fallback mappings when address data is split across `mailing_*`, `primary_*`, or alternate families like `address_*`.

          
After running, check `run_summary.json`, the review bucket size, and exact-ID overlap on unmatched rows when both sides have a meaningful shared ID.
